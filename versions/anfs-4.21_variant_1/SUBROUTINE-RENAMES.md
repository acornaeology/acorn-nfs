# Subroutine and label rename candidates

Running list of label / subroutine names that don't match what the routine
actually does in 4.21_variant_1. Most are stale carry-over from earlier
versions where the address held different code, or generic placeholders
the original sweep never refined.

These are deferred until after the inline-comment pass (the
[INTERPOLATED-GAP-CLEANUP.md](INTERPOLATED-GAP-CLEANUP.md) workflow)
completes, so we name routines based on a confirmed understanding of what
they do rather than inheriting the existing fiction.

## Procedure (when ready to act)

1. For each entry, confirm the proposed name against the current code +
   inline comments.
2. Rename via `uv run fantasm labels rename <old> <new>` if available, or
   edit the `label(...)` / `subroutine(...)` declaration directly.
3. Search for stale references in descriptions / comments and update.
4. `fantasm verify` + `fantasm lint` after each rename.

## Notes

- When renaming, prefer single descriptive names over compound `_sub_`-style
  hierarchies; the latter were a heuristic from the original autogenerated
  pass and don't reflect actual code structure.
- Some candidates may turn out to be correct after a wider review (e.g. if
  `print_dir_syntax` is in fact only used for *Dir-related paths in 4.21
  even though it currently also serves *FS/*PS, that may be an accident of
  the help-string layout rather than intent). Verify before renaming.
