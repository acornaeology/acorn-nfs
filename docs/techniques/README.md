# Disassembly recovery techniques

When a new ROM version is generated by mapping addresses from a known version
through `difflib.SequenceMatcher`, a substantial fraction of the source's
annotations end up *unmapped* — the LCS algorithm could not locate the
corresponding bytes in the new version. The papers in this directory describe
techniques developed during the ANFS 4.18 → 4.21 (variant 1) port for
**recovering** those unmapped routines: pinning each one to its true new
address so that the carry-over comments and descriptions can be re-attached.

The four primary techniques are complementary. Pick the right one based on
how the routine is reached at runtime:

| Routine reached via | Best technique |
|---|---|
| Direct JSR/JMP from a routine that *did* map | [02-jsr-following.md](02-jsr-following.md) |
| Inline string preceding a `JSR print_inline` | [03-string-anchoring.md](03-string-anchoring.md) |
| PHA/PHA/RTS dispatch from a sub-function table | [04-dispatch-table-decoding.md](04-dispatch-table-decoding.md) |
| Anything else (general fallback) | [01-opcode-fingerprinting.md](01-opcode-fingerprinting.md) |

For the underlying algorithmic foundation that the supplementary block-matcher
uses, see [00-seed-and-extend.md](00-seed-and-extend.md).

The accompanying tools live in `src/disasm_tools/`:

- `blockmatch.py` — primary LCS + supplementary seed-and-extend
- `fingerprint.py` — sliding-window single-routine fingerprint match
- `mos6502.py` — opcode length / mnemonic tables (6502 and 65C02)

All techniques produce candidate addresses; **always verify** by spot-checking
the prologue opcodes against the source version before declaring the
subroutine. Several techniques (especially fingerprinting on long routines)
will produce occasional false positives where a body sub-sequence matches
strongly but the entry-point prologue diverges.
